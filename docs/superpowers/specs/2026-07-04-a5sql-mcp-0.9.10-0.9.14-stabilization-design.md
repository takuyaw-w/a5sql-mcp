# a5sql-mcp 0.9.10-0.9.14 Stabilization Roadmap Design

## Goal

1.0.0 の実装に進む前に、現在の `0.9.10: 1.0.0 Preflight RC` を 5 つの小さな安定化 milestone に分解する。0.9.10 以降は新機能追加ではなく、公開 contract、出力 contract、published package 検証、docs / onboarding、final dry run を順番に固定し、1.0.0 で行う作業を version bump、tag、publish に近づける。

## Context

0.9.9 では API Freeze Rehearsal として、`.a5er` 起動時の 22 tool、stable read-only / experimental draft の分類、代表 draft disclosure、README / AGENTS.md / `.agents/skills/a5sql-mcp/SKILL.md` の分類を固定した。現在の `main` は root、parser、core、cli、MCP server metadata が `0.9.9` で揃っている。

現行 ROADMAP の `0.9.10: 1.0.0 Preflight RC` は、`release:check`、`published:check`、MCP startup、`tools/list`、代表 `callTool`、adversarial fixture、docs、package metadata、version、tag preflight、1.0.0 non-goal の確認をまとめて扱っている。これは final preflight としては妥当だが、1 milestone としては大きく、失敗時に contract 問題、package 問題、docs 問題、release 手順問題を切り分けにくい。

過去の release candidate 作業では、published-style MCP startup、version lockstep、tag/version mismatch、`Tools: (none)`、docs/tool 一覧 drift が実際に問題になった。1.0.0 前は、それらを一度に確認するのではなく、小さい milestone と検証単位へ分ける。

## Scope

対象にするもの:

- `ROADMAP.md` の `0.9.10: 1.0.0 Preflight RC` を `0.9.10` から `0.9.14` の 5 milestone に分解する。
- `0.9.10` を API surface / docs / test の contract audit にする。
- `0.9.11` を representative structuredContent と trusted metadata / untrusted payload の output contract audit にする。
- `0.9.12` を installed package と MCP client 経由の validation matrix にする。
- `0.9.13` を README / AGENTS.md / skill の docs and onboarding freeze にする。
- `0.9.14` を 1.0.0 直前の final RC dry run にする。
- `1.0.0` の milestone と優先順位を、5 milestone の後に自然につながる表現へ調整する。

対象外:

- 0.9.10 以降の実装をこの変更で行うこと。
- package version を `0.9.10` 以上へ bump すること。
- MCP tool の追加、削除、rename。
- output field name の変更。
- release tag、push、publish。
- DB 接続、SQL 実行、A5:SQL 設定ファイルへの書き込み、資格情報の復号や表示。

## Milestone Design

### 0.9.10: Preflight Contract Audit

0.9.9 で凍結した API surface が、実装、README、AGENTS.md、skill、test で本当に一致しているかを再監査する。`tools/list` を source of truth とし、tool 名、description、input schema、stable read-only / experimental draft 分類、docs の tool 一覧 drift を確認する。ここでは runtime behavior を変えることより、1.0.0 前に破壊的変更すべき API 名や分類が残っていないことを確認する。

### 0.9.11: Output Contract Consistency Audit

代表 tool の `structuredContent` を横断的に確認する。`found`、`truncated`、`hasMore`、`warnings`、`nextAction`、`contentIsUntrusted`、`trustedMetadataFields`、`untrustedPayloadFields`、`draftOutputFields` の使い方を揃える。A5:SQL 由来の payload が `warnings`、`message`、`code`、`nextAction` のような trusted metadata に混ざらないことを重視する。既存 JSON contract を不必要に変えず、必要な場合は regression test で意図を固定してから最小修正する。

### 0.9.12: Published Package / MCP Client Matrix

source tree の直接実行ではなく、install 後の package と MCP client 経由での動作確認を広げる。`published:check` の assertion を見直し、tarball install 後の CLI bin から MCP server を起動して、`tools/list`、代表 `callTool`、root required、secret masking、untrusted content signal、draft disclosure、最低限の adversarial assertion を確認する。registry や network への依存を増やしすぎず、local tarball install の再現性を優先する。

### 0.9.13: Docs / Onboarding Freeze

README、AGENTS.md、`.agents/skills/a5sql-mcp/SKILL.md` の利用者向け・エージェント向け説明を最終固定する。install、起動、`roots` / `A5SQL_MCP_ROOTS`、安全な範囲読み取り、`contentIsUntrusted`、trusted metadata、draft output、1.0.0 non-goal を同じ境界で説明する。生成補助 tool は review 用 draft であり、SQL 実行、migration 適用、ファイル書き込みを行わないことを明確にする。

### 0.9.14: Final RC Dry Run

1.0.0 として tag / publish する直前の final rehearsal にする。`rtk pnpm release:check`、`rtk pnpm published:check`、version lockstep、tag preflight、npm publish workflow 前提、release note 相当の整理を確認する。ここでは新機能や大きな contract 変更を入れず、失敗が見つかった場合は必要に応じて `0.9.15` を切る判断にする。

## Design

`ROADMAP.md` の過去 milestone はそのまま残し、現在の `0.9.10: 1.0.0 Preflight RC` 節だけを 5 milestone に置き換える。これにより、0.9.0 から 0.9.9 までの履歴と完了条件は保持しつつ、次の作業単位だけを細分化する。

各 milestone は `目的`、`主な作業`、`完了条件` を持つ既存形式に合わせる。`0.9.14` の完了条件には「1.0.0 でやることが version bump / tag / publish に近い状態」を明記し、修正が必要な場合は新しい patch milestone を切れる余地を残す。

優先順位の節は、最優先項目を過去の 0.9.2-0.9.9 だけでなく、これからの 0.9.10-0.9.14 に対応する表現へ更新する。新しい ORM 対応、DB 実行、書き込み系 tool、UI は引き続き後回しにする。

## Testing

この変更自体は ROADMAP 文書の再編であり、runtime code は変更しない。検証は次を行う。

- `rg` で `0.9.10`、`0.9.11`、`0.9.12`、`0.9.13`、`0.9.14`、`1.0.0` の節が ROADMAP 上で期待通り並ぶことを確認する。
- `rtk pnpm agent:preflight` を implementation plan の Task 0 として実行し、実装時は main 直接編集を避ける。
- 文書更新後は `rtk pnpm release:check` を実行する。ROADMAP だけの変更でも、format / lint が docs を見る可能性を考えて release gate で確認する。

## Success Criteria

- `ROADMAP.md` の `0.9.10` が単一の大きな preflight ではなく、`0.9.10` から `0.9.14` の 5 milestone に分解されている。
- 5 milestone が contract、output、published package、docs、final RC の順に並び、1.0.0 前の安定化作業として独立して実装しやすい。
- 各 milestone の目的、主な作業、完了条件が、既存 ROADMAP の文体と粒度に合っている。
- 1.0.0 の include / non-goal 方針は変えず、DB 接続、SQL 実行、書き込み、資格情報復号・表示は引き続き対象外である。
- この design は ROADMAP 再編だけを扱い、0.9.10 以降の実装や version bump に踏み込まない。
