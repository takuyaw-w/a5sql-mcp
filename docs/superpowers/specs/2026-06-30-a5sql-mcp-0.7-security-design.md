# a5sql-mcp 0.7.0 Security / Privacy Hardening Design

Date: 2026-06-30
Status: Approved

## Goal

Implement the ROADMAP.md 0.7.0 security and privacy hardening scope for a local, read-only MCP server that may inspect A5:SQL Mk-2 files containing credentials or user-specific paths.

0.7.0 should make the existing public surface safer and better documented without adding database execution, credential decryption, file writes, or new MCP tools.

## Scope

In scope:

- Expand secret masking test coverage and add only the minimum masking implementation needed for those cases.
- Confirm that `read_a5sql_file`, `read_a5sql_asset`, and `search_a5sql_assets` return masked body text and snippets.
- Recheck the `list_a5sql_connections` public contract so secret values, connection strings, and internal `sourcePath` values are not returned by the MCP response.
- Confirm error and warning responses do not expose raw secret values, raw file content, or unnecessary long absolute paths.
- Update README, AGENTS.md, and `.agents/skills/a5sql-mcp/SKILL.md` so read-only behavior, least-privilege roots, and the database execution non-goal are consistent.
- Bump the repo version metadata to 0.7.0 in lockstep.

Out of scope:

- Connecting to a real database.
- Executing SQL against a real database.
- Decrypting or displaying credentials.
- Writing, modifying, or deleting local A5:SQL assets.
- Adding new MCP tools.
- Removing the existing successful-response asset `path` metadata. That is a compatibility-sensitive contract change and should be designed separately if needed.
- Adding a repository-wide static secret scanner. This can be considered later, but 0.7.0 focuses on runtime response contracts and targeted tests.

## Current Contract

The MCP server is read-only. It can inspect local files and configured roots, but it does not connect to databases and does not mutate the filesystem.

The public tools relevant to 0.7.0 are:

- `read_a5sql_file`: reads the startup file content with range limits.
- `read_a5sql_asset`: reads a discovered or explicit asset with root constraints and size limits.
- `search_a5sql_assets`: finds A5:SQL related assets under roots and returns masked snippets.
- `list_a5sql_connections`: extracts connection candidates from A5:SQL config roots without returning secrets.

For discovered assets, the existing successful response may include path metadata. 0.7.0 does not remove that field. Instead, it documents least-privilege roots and tests that response content, snippets, error details, and warning details do not introduce secret leaks.

## Architecture

Secret masking remains centralized in `packages/core/src/mask.ts`.

Asset and connection code should not grow separate ad hoc masking rules unless there is a clear boundary-specific reason. The default path is:

1. Strengthen `maskSensitiveText`.
2. Keep `packages/core/src/assets.ts` and `packages/core/src/connections.ts` using the shared masking behavior.
3. Keep `packages/cli/src/mcp/tool-handlers.ts` responsible for shaping public MCP responses and omitting internal-only fields.

The implementation should preserve existing schema shape where possible. Tests should drive any code changes.

## Data Flow

### read_a5sql_file

The startup file is read, decoded, masked, and then returned according to `offsetChars` or `startLine` / `maxLines`.

Expected 0.7.0 property:

- Returned text does not contain raw passwords, tokens, API keys, private keys, authorization credentials, or connection-string credentials.

### read_a5sql_asset and search_a5sql_assets

Assets are resolved under configured roots or explicit allowed paths, then read with existing file-type and size checks.

Expected 0.7.0 properties:

- `read_a5sql_asset` content is masked.
- `search_a5sql_assets` snippets are masked.
- Unsupported, binary, missing, out-of-root, or no-root responses do not include raw file content or raw secret values.
- Existing success response path metadata is preserved for compatibility.

### list_a5sql_connections

Connection candidates are extracted from masked text and then shaped for MCP output.

Expected 0.7.0 properties:

- Secret values are not returned.
- Full connection strings containing credentials are not returned.
- `sourcePath` remains internal and is not exposed by the MCP response.
- `revealNonSecret: true` can reveal non-secret host/database/user-style fields only where the contract allows it, and must still not reveal passwords, tokens, or credential-bearing URLs.

## Error and Warning Policy

Error and warning responses should help the caller recover without exposing local secrets.

0.7.0 should prefer:

- Stable error codes.
- Short human-readable messages.
- A `nextAction` or equivalent recovery hint where already used by the codebase.
- No raw file content.
- No raw credential values.
- No newly added long absolute paths in failure details.

Successful asset discovery can still expose path metadata according to the existing contract. This is controlled by least-privilege root configuration rather than silently changing response shape in a patch-like hardening release.

## Documentation Updates

README should make these points explicit for users:

- Configure `A5SQL_MCP_ROOTS` to the narrowest directories needed for the current task.
- Avoid using broad home, drive, or project-parent roots unless the caller intentionally wants that discovery scope.
- The server is read-only for local files and does not execute SQL.
- Real DB connection and SQL execution are not part of the 1.0.0 scope.

AGENTS.md should remain the developer/agent contract:

- Preserve read-only defaults.
- Do not return secrets in responses, logs, fixtures, or docs.
- Do not add DB execution without a separate explicit design.

`.agents/skills/a5sql-mcp/SKILL.md` should align with the current tool surface and 0.7.0 safety expectations so future Codex work starts from the same contract.

## Test Plan

Use TDD. Add failing tests first, then make the minimum implementation changes required.

Planned test coverage:

- `packages/core/test/mask.test.ts`
  - JSON-style quoted keys such as `"password": "..."`.
  - env/export forms such as `export API_KEY=...`.
  - Authorization Bearer and Basic credentials.
  - URL query credentials such as `?password=...` or `&token=...`.
  - ODBC/JDBC-style credential-bearing connection strings.
- `packages/core/test/assets.test.ts`
  - Asset read and search snippets do not contain raw secret values.
- `packages/cli/test/mcp-asset-tools.test.ts`
  - `read_a5sql_asset` and `search_a5sql_assets` MCP responses are masked.
  - no-roots, out-of-root, and missing-asset errors avoid raw content, raw secrets, and unnecessary absolute paths.
  - `list_a5sql_connections` default and `revealNonSecret` responses do not expose secrets or internal `sourcePath`.
- `packages/cli/test/mcp-server-smoke.test.ts`
  - smoke coverage remains aligned with the 0.7.0 version and no-secret response expectations.

Final verification:

- `rtk pnpm release:check`
