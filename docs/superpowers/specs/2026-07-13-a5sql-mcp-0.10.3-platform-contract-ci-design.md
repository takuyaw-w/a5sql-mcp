# a5sql-mcp 0.10.3 Platform, Draft, Public Contract and CI Hardening Design

Status: approved

## 背景

`ROADMAP.md` の 0.10.3 は、0.10.2 で回復した contract integrity と parser fidelity を前提に、OS / encoding 差分、model draft、MCP output contract、CLI / CI / publish 境界を 1.0.0 向けに固定する milestone である。

現行実装には次の gap がある。

- CLI と core にほぼ同じ text decoder が重複し、短い UTF-16LE と binary の判定条件、read metadata が一致していない。
- location candidate の path dedupe が全 platform で lowercase 化され、Linux の case-sensitive path を誤って同一扱いする。
- Laravel / SQLAlchemy model draft が A5:ER 由来の table / column / relation 名を identifier や string literal に直接埋め込み、予約語、先頭数字、記号、非 ASCII、quote、backslash で無効な code を生成し得る。
- `list_a5sql_connections` は返却件数を `totalConnectionCount` として返すため、探索全体の total と誤認させる。
- stable read-only tool に MCP `outputSchema` がなく、success / error / unrecognized / truncated variant を client が機械的に検証できない。
- CLI 直実行は `parseFile()` の raw result をそのまま stdout へ出し、MCP 側の masking policy と一致しない。
- GitHub Actions は tag publish workflow だけで、PR / main、Windows、publish 前 installed-package validation が自動化されていない。
- ROADMAP の「現在の主な弱み」が 0.10.2 完了後の実装状態を反映していない。

`ctx` では 0.10.3 固有の過去判断を取得できなかったため、現行 `ROADMAP.md`、コード、test、workflow を source of truth とする。

## 目的

- 起動時ファイルと asset 読み取りで同じ decoder と metadata contract を使う。
- path case sensitivity を platform contract として明示する。
- unusual / hostile identifier から構文上安全な Laravel / SQLAlchemy draft を生成する。
- connection total が正確か不明かを client が判断できるようにする。
- stable read-only tool の公開 output contract を MCP `outputSchema` で固定する。
- CLI 直実行の default output を mask 済みにする。
- PR / main / tag publish の各段階で portability と installed package regression を検出する。

## Architecture 上の位置づけ

Primary pattern は read-only Resource Gateway とし、A5:SQL / A5:ER の domain data を LLM 向けに変換する Domain-Specific Adapter の性質を維持する。

0.10.3 は新しい capability を増やす milestone ではない。既存 22 tool の portability と public contract を固める。次の anti-pattern を避ける。

- decoder や masking が入口ごとに分岐する unsanitized resource content。
- invalid identifier を含む draft を検証 signal なしで返すこと。
- count が不明なのに total を推測する曖昧な metadata。
- `outputSchema` を全 field optional の空 schema にして contract 固定を装うこと。
- CI だけで成立し、local `release:check` / `published:check` と意味がずれる検証。

## 設計

### 1. Canonical text decoder

`packages/core/src/text.ts` を decoder の唯一の実装にする。CLI の `packages/cli/src/text.ts` は core API を再公開する compatibility adapter にする。

canonical API は次を返す。

- `text`
- `encoding`
- `bytesRead`
- `sizeBytes`
- `truncated`

判定順序:

1. UTF-8 BOM
2. UTF-16LE BOM
3. 短い UTF-16LE heuristic
4. binary
5. strict UTF-8
6. Shift_JIS
7. UTF-16LE
8. UTF-8 lossy fallback

短い BOM なし UTF-16LE は、4 byte 以上、偶数長、複数の odd-position NUL、even-position NUL の少なさ、printable non-NUL 比率を組み合わせる。`[0x01, 0x00, 0x02, 0x03]` のような tiny binary は UTF-16LE にしない。

source が byte 上限で切断されている場合、strict UTF-8 decoder の streaming mode で末尾の未完 multibyte だけを許容する。内部の不正 byte sequence は許容しない。

共通 fixture は CLI と core の両入口へ適用する。

- empty
- UTF-8
- UTF-8 BOM
- Shift_JIS
- UTF-16LE BOM
- short UTF-16LE without BOM
- binary
- tiny binary with odd-position NUL
- truncated UTF-8 multibyte

### 2. Platform-aware path dedupe

path dedupe key は `path.resolve()` 後に platform ごとに決める。

- `win32`: lowercase key
- それ以外: resolved path をそのまま key にする

`detectA5sqlLocations()` の `options.platform` を dedupe に渡す。Linux 上でも `platform: "win32"` の pure test で Windows contract を検証する。

Wine path は Linux filesystem 上の path として case-sensitive に扱う。realpath、root containment、symlink policy は変更しない。

### 3. Language-specific draft safety

model draft generator に、言語別の identifier encoder と string literal encoder を追加する。

PHP:

- class name、relation method name を有効な PHP identifier に正規化する。
- PHP reserved word、先頭数字、空 identifier を安全な prefix / suffix で回避する。
- table、column、foreign/local key、fillable、casts は PHP string literal encoder を通す。
- quote、backslash、control character、`$` を意図しない補間なしで保持する。

Python:

- class name、attribute name を有効な Python identifier に正規化する。
- Python keyword、先頭数字、空 identifier を回避する。
- attribute name と実 column 名が異なる場合は `mapped_column("physical name", ...)` を使う。
- table / foreign key は Python string literal encoder を通す。

共通:

- 正規化後の identifier collision を deterministic suffix で解消する。
- 元の physical name は string literal metadata として保持する。
- non-ASCII だけで構成された名前も deterministic fallback identifier を持つ。
- generator は `syntaxValidation` metadata を返し、安全 template / identifier / literal validation を通過したか示す。
- test / CI では生成した PHP を `php -l`、Python を `python -m py_compile` で検証する。
- compile test 用 runtime がない local `release:check` を壊さず、CI では runtime setup 後に dedicated syntax check を必須にする。

### 4. Connection count semantics

core に metadata 付き `listA5sqlConnectionsWithMetadata()` を追加し、既存 `listA5sqlConnections()` は returned connections だけを返す compatibility wrapper として残す。

metadata:

- `connections`
- `knownConnectionCount`
- `totalConnectionCount: number | null`
- `totalConnectionCountIsExact`
- `returnedConnectionCount`
- `truncated`
- `visitedFileCount`
- `cutoffReason`

探索対象 asset の走査が完了した場合だけ exact total を返す。asset limit / max files で探索が打ち切られた場合は `totalConnectionCount: null` とし、`knownConnectionCount` と cutoff metadata を返す。

MCP handler は client の `limit` を返却上限として使い、total 検出のために `limit + 1` だけを取得する既存実装を廃止する。

### 5. Stable tool outputSchema

次の stable read-only 17 tool に MCP `outputSchema` を登録する。

- `describe_a5sql_file`
- `parse_a5sql_file`
- `read_a5sql_file`
- `detect_a5sql_locations`
- `read_a5sql_asset`
- `list_a5sql_connections`
- `search_a5sql_assets`
- `parse_a5sql_asset`
- `list_a5sql_tables`
- `describe_a5sql_table`
- `explain_a5sql_table`
- `list_a5sql_relationships`
- `find_a5sql_tables`
- `find_a5sql_columns`
- `review_a5sql_schema`
- `suggest_schema_changes`
- `compare_a5er_with_live_schema`

experimental draft 5 tool には 0.10.3 で stable output schema を追加しない。

schema は reusable fragment と tool-specific variant で構成する。

- trusted / source / untrusted metadata fragment
- structured error fragment
- unrecognized A5:ER fragment
- truncation / paging fragment
- tool-specific success payload

top-levelの意味が固定済みの field は required にする。A5:ER table / column や issue detail のように将来の additive extension を許す nested payload だけを extensible にする。空の passthrough schema は使用しない。

MCP client test は次を確認する。

- `tools/list` の stable tool に `outputSchema` がある。
- experimental draft tool に stable output schema がない。
- success / error / unrecognized / truncated の代表 response が schema に適合する。
- `JSON.parse(content[0].text)` と `structuredContent` が deep equal になる。

### 6. CLI direct output masking

`a5sql-mcp <file>` は mask 済み JSON を default にする。`parseFile()` 自体は MCP 内部でも使うため raw parse result を維持し、CLI stdout 境界で recursive masking を行う。

CLI:

```text
a5sql-mcp <file>
a5sql-mcp <file> --unsafe-raw-output
```

default:

- secret-like key の値を mask する。
- string payload 内の password、token、URL userinfo、connection string、private key を mask する。
- stdout は JSON だけにする。

`--unsafe-raw-output`:

- 生の parse result を stdout へ出す。
- stderr に固定警告を出す。
- MCP mode では受け付けない。
- option 名を docs と help で危険性が明確な形にする。

### 7. PR / main CI

`.github/workflows/ci.yml` を追加する。

trigger:

- pull request
- `main` push

matrix:

- Ubuntu
- Windows
- Node.js 24

必須 command:

- `pnpm format:check`
- `pnpm lint`
- `pnpm build`
- `pnpm test`
- `pnpm typecheck`

Python と PHP を setup し、dedicated model draft syntax check で `python -m py_compile` と `php -l` を実行する。Windows path / decoder fixture は matrix の通常 test に含める。

### 8. Publish boundary

tag publish workflow は次の順序にする。

1. install
2. tag / version lockstep
3. `pnpm release:check`
4. `pnpm published:check`
5. npm publish

installed package bin の MCP startup、`tools/list`、representative client matrix が失敗した場合は publish しない。

### 9. Version、docs、ROADMAP

- root、parser、core、CLI、MCP server metadata を `0.10.3` に揃える。
- README、AGENTS.md、repo-local skill に decoder、CLI masking、output schema、CI / publish boundary を記載する。
- docs redaction / contract drift test を更新する。
- ROADMAP 0.10.3 を完了にする。
- 「現在の主な弱み」から 0.10.2 / 0.10.3 で解消した記述を除去または現在形へ更新する。

## Test 方針

TDD で次を固定する。

- CLI / core decoder fixture parity。
- Linux case-sensitive / Windows case-insensitive path dedupe。
- PHP / Python reserved word、先頭数字、記号、quote、backslash、非 ASCII、collision。
- generated PHP / Python syntax compile。
- exact / unknown connection total と cutoff metadata。
- stable 17 tool の `outputSchema` presence と representative variant validation。
- `content[0].text` / `structuredContent` parity。
- CLI safe default と explicit unsafe warning。
- Ubuntu / Windows workflow と publish 前 `published:check`。
- installed package bin の代表 contract。

## 非目的

- tool 数、tool profile、既定 22 tool surface の変更。
- MCP Resource の追加。
- experimental draft の卒業。
- DB 接続、SQL 実行、ファイルシステム書き込み tool。
- 新しい ORM / migration framework。
- `tool-outputs.ts` の無関係な大規模分割。
- remote transport、authentication、stateful session。

## 完了条件

- decoder / path / draft generator の platform contract が test で固定される。
- connection count と stable tool output contract を MCP client が機械的に検証できる。
- CLI default output が mask 済みで、raw output が明示 opt-in になる。
- PR / main の Ubuntu / Windows CI と tag publish 前 installed-package check が定義される。
- `rtk pnpm release:check`、`rtk pnpm published:check`、`rtk pnpm audit --prod` が通る。
- version / docs / ROADMAP が 0.10.3 と一致する。

## 承認事項

この design は、canonical core decoder、platform-aware path dedupe、language-specific safe model draft、nullable exact connection total、stable 17 tool outputSchema、safe-by-default CLI、Ubuntu / Windows CI、publish 前 installed-package validation の方針で承認された。
