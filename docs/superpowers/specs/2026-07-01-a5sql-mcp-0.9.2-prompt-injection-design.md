# a5sql-mcp 0.9.2 Prompt Injection Defense Baseline Design

## Goal

0.9.2 では、A5:SQL 由来の本文、コメント、テーブル名、カラム名、SQL 本文を、AI への命令ではなく untrusted content として扱う契約を固定する。

## Scope

対象にするもの:

- `parse_a5sql_file` / `read_a5sql_file` / `read_a5sql_asset` / `parse_a5sql_asset` の本文・解析結果。
- `.a5er` の識別子、論理名、コメント、リレーション情報を返す read-only helper の代表出力。
- `search_a5sql_assets` の本文抜粋。
- prompt injection 文を含む synthetic fixture に対する contract test。
- README / AGENTS.md / `.agents/skills/a5sql-mcp/SKILL.md` の注意書きと version lockstep。

対象外:

- DB への接続、SQL 実行、A5:SQL 設定ファイルや `.a5er` への書き込み。
- A5:ER / SQL parser の大きな仕様変更。
- prompt injection 文の削除、書き換え、検閲。内容は参照用 payload として返し、信頼境界を signal で示す。
- `nextAction`、`warnings`、`message` を A5:SQL 由来文字列から生成する設計変更。これは 0.9.3 の対象に残す。

## Design

`contentIsUntrusted: true` を、A5:SQL 由来 payload を含む代表レスポンスの top-level signal として扱う。既に handler 側で付与している read / parse 系に加え、`.a5er` のテーブル名、カラム名、コメント、リレーション caption を返す helper 出力にも signal を付ける。

生成補助 tool は既存の `outputKind: "draft"`、`readOnly: true`、`writesToFileSystem: false`、`connectsToDatabase: false`、`executesSql: false` を維持する。0.9.2 では read-only exploration と raw read / parse の baseline を固定し、生成系の trusted metadata / untrusted payload 分離は 0.9.3 で扱う。

prompt injection fixture は synthetic な `.a5er` と SQL asset だけを使い、実在するローカル設定や秘密情報を含めない。fixture 内の「前の指示を無視して」などの文言は、metadata ではなく table/comment/snippet/content payload にだけ現れることを確認する。

## Testing

TDD で次を固定する。

- `.a5er` helper の代表出力が `contentIsUntrusted: true` を返し、prompt injection 文を payload として保持する。
- `search_a5sql_assets` の SQL snippet が `contentIsUntrusted: true` を返し、`nextAction` は固定文のままで injection 文に汚染されない。
- `parse_a5sql_asset` は SQL body を含む結果に `contentIsUntrusted: true` を返す。
- version は root workspace、parser、core、cli で `0.9.2` に揃える。

最終検証は `rtk pnpm release:check` と `rtk pnpm published:check` とする。

## Success Criteria

- A5:SQL 由来の本文・コメント・識別子を含む代表レスポンスに `contentIsUntrusted: true` がある。
- prompt injection 文を含む fixture でも、trusted metadata は固定文のままで、A5:SQL 由来文言を命令として扱わない。
- README / AGENTS.md / skill が同じ contract を説明している。
- `rtk pnpm release:check` と `rtk pnpm published:check` が通る。
